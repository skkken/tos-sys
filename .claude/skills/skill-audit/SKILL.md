# スキル監査・改善スキル

既存のClaude Codeスキルを5つの評価軸で監査し、合格基準を満たさない場合は改善を実施する。

評価基準の詳細は `${CLAUDE_SKILL_DIR}/reference.md` を参照。
監査レポートは `${CLAUDE_SKILL_DIR}/templates/report.md` のテンプレートに従う。

## 入力

ユーザーから以下を受け取る:

- **対象スキル**: スキル名またはスキルディレクトリのパス
- **モード**（任意）: `check`（監査のみ）または `improve`（監査+改善）。デフォルトは `improve`

スキル名のみ指定された場合は `.claude/skills/{スキル名}/` をパスとして使用する。

## Agent Teams活用方針

本スキルでは3名の監査エージェントがAgent Teamsとして協働する。各auditorは独立に監査した後、発見事項を互いに共有し、横断的な問題を捕捉する。

- **TeamCreate**: 監査チームを作成し、全エージェントを所属させる
- **TaskCreate/TaskUpdate**: 各監査タスクの進捗をタスクボードで追跡する
- **SendMessage**:
  - エージェント間の発見事項共有に使用する（初期監査後の共有フェーズ）
  - 最終評価のlead宛報告に使用する
  - フロー制御には使わない

### データ受け渡しのルール
- 各エージェントのプロンプトには必要な情報を全てインライン展開する
- 終了済みエージェントへのSendMessageは行わない
- TaskのblockedByはメタデータとして設定し、フロー制御はオーケストレーターが担う

## ワークフロー

### ステップ1: 対象スキルの全ファイル読み込み

対象スキルディレクトリ内の全ファイルを `Glob` で列挙し、`Read` で全て読み込む。
SKILL.md、参照ファイル、テンプレート等の構成を把握する。

### ステップ2: チーム・タスクの準備

#### チーム作成

```
TeamCreate: team_name="skill-audit-{対象スキル名}"
```

#### タスク作成

以下のタスクを `TaskCreate` で作成する:

1. **技術的正確性の監査** — テクニカル・オーディター担当
2. **プロセス・役割設計の監査** — プロセス・デザイナー担当
3. **コンテンツ・プロンプト品質の監査** — コンテンツ・ベリファイア担当
4. **結果統合・判定** — オーケストレーター（lead）担当
5. **改善実施**（improveモードの場合のみ） — オーケストレーター（lead）担当

タスク4は `TaskUpdate` で `addBlockedBy` にタスク1,2,3を設定する。
タスク5は `TaskUpdate` で `addBlockedBy` にタスク4を設定する。

### ステップ3: 並行監査（3エージェント同時起動）

タスク1,2,3をそれぞれ `TaskUpdate` で `in_progress` に更新してから、以下の3エージェントを**1つのメッセージで並行起動**する。

各エージェントのプロンプトには以下を**必ず含める**:
- チーム名（team_name）
- 担当タスクID
- 対象スキルの全ファイル内容（SKILL.md、reference.md、examples.md、templates等の内容そのもの）
- `${CLAUDE_SKILL_DIR}/reference.md` の該当する評価基準セクションの内容

各エージェントの完了時の指示:
- `TaskUpdate` で担当タスクを `completed` に更新する
- `SendMessage` で `lead` 宛（`to: "lead"`）に評価結果を送信する

#### エージェント1: テクニカル・オーディター

```
Agent:
  name: "technical-auditor"
  team_name: "skill-audit-{対象スキル名}"
  subagent_type: "general-purpose"
  prompt: |
    あなたはテクニカル・オーディターです。スキル定義がClaude Codeの実際のAPI・ツール仕様と整合しているかを検証します。

    ## チーム情報
    - チーム名: skill-audit-{対象スキル名}
    - あなたの名前: technical-auditor
    - 担当タスクID: {タスク1のID}

    ## 対象スキルのファイル内容
    {対象スキルの全ファイル内容をここに貼り付ける}

    ## 評価基準
    {reference.mdの「1. 技術的正確性」セクションをここにインライン展開する}

    ## 作業内容
    1. スキルが使用する全ツール（Agent, SendMessage, TaskCreate等）がallowed-toolsに含まれているか確認
    2. 各ツールの使い方がClaude Codeの実際の挙動と一致するか検証
       - Agentツール: 戻り値で結果を受け取る仕組みになっているか
       - SendMessage: 使用している場合、受信側のエージェントが稼働中か
       - TaskCreate/TaskUpdate: 使用目的が明確か（可視化/メタデータとして適切か、フロー制御に誤用されていないか）
       - WebSearch/WebFetch: 調査対象が実際にWeb検索で確認可能か
    3. エージェントのプロンプトにファイルパス参照ではなく内容がインライン展開されるよう指示されているか
    4. 存在しない機能・パラメータへの依存がないか

    ## 出力形式
    以下の形式で出力してください:
    - スコア: 1-5（評価基準に基づく）
    - 発見した問題: 各問題について「致命的/重大/軽微」の重要度と具体的な説明
    - 改善提案: 各問題の具体的な修正案

    ## 対話プロセス（重要）

    ### フェーズA: 初期監査
    上記の作業内容に従い、独立に監査を実施する。

    ### フェーズB: 発見事項の共有
    初期監査が完了したら:
    1. 発見した問題（特に重大・致命的なもの）をprocess-designerとcontent-verifier宛にSendMessageで共有する
    2. 他のauditorからの発見事項メッセージを確認する
    3. 他のauditorの発見が自分の評価軸に影響する場合、スコアや指摘事項を調整する
       - 例: process-designerが「ワークフローに冗長なステップがある」→ 関連するツール使用にも問題がないか再確認
       - 例: content-verifierが「プロンプトにツール名の誤記がある」→ allowed-toolsとの整合性を再確認

    ### フェーズC: 最終報告
    4. 調整後の最終評価をSendMessageでlead宛に送信する
    5. TaskUpdateで担当タスク（ID: {タスク1のID}）をcompletedに更新する
```

### ステップ4: 結果統合と判定

3エージェントの戻り値を受け取ったら、タスク4を `TaskUpdate` で `in_progress` に更新する。

3エージェントの結果を統合し、以下の基準で合否を判定する。

#### スコア集計（5軸 × 5点満点 = 25点満点）

| # | 評価軸 | 担当エージェント |
|:--|:--|:--|
| 1 | 技術的正確性 | テクニカル・オーディター |
| 2 | プロセス設計 | プロセス・デザイナー |
| 3 | 役割設計 | プロセス・デザイナー |
| 4 | コンテンツ正確性 | コンテンツ・ベリファイア |
| 5 | プロンプト品質 | コンテンツ・ベリファイア |

#### 合格基準

以下の**全て**を満たす場合に合格（PASS）:

- **合計スコアが20点以上**（25点満点）
- **全軸が3点以上**（1軸でも2点以下があれば不合格）
- **致命的な問題がゼロ**

#### 判定結果の出力

`${CLAUDE_SKILL_DIR}/templates/report.md` のテンプレートに沿って監査レポートを作成し、ユーザーに提示する。

タスク4を `TaskUpdate` で `completed` に更新する。

### ステップ5: 判定に基づく分岐

#### PASS の場合

監査レポートを提示して終了。軽微な改善提案がある場合は付記する。
`TeamDelete` でチームを削除する。

#### FAIL の場合

##### checkモード
監査レポートを提示し、不合格の理由と改善提案を一覧で示して終了。
`TeamDelete` でチームを削除する。

##### improveモード（デフォルト）
1. 監査レポートをユーザーに提示する
2. 発見された問題と改善案をユーザーに示し、修正方針の合意を得る（**ユーザーチェックポイント**）
3. 合意が得られたら、タスク5を `TaskUpdate` で `in_progress` に更新し、オーケストレーター自身がスキルファイルを修正する
   - 致命的な問題から順に修正
   - 各ファイルを `Read` → `Edit` で修正
4. タスク5を `TaskUpdate` で `completed` に更新する
5. 修正完了後、**再監査を実施**（新たにタスクを作成し、ステップ3に戻る）
6. PASSになるまで繰り返す（最大2回まで。2回改善してもFAILの場合はユーザーに報告して判断を仰ぐ）
7. 全作業完了後、`TeamDelete` でチームを削除する

## 重要な注意事項

- 3エージェントは**必ず1つのメッセージで同時起動**すること
- 各エージェントは初期監査後に**他の2名にSendMessageで発見事項を共有**すること
- 他のauditorの発見を受けて**スコア・指摘事項を調整してからlead宛に最終報告**すること
- 各エージェントのプロンプトには**チーム名・自分の名前・担当タスクID**を含めること
- 各エージェントのプロンプトには対象スキルの**全ファイル内容**とreference.mdの評価基準を**インライン展開**すること
- 改善時はユーザーの合意なく修正を開始しないこと
- 再監査は最大2回までとし、無限ループを防止すること
- 全作業完了後は必ず `TeamDelete` でチームを削除すること
